## CRITICAL findings

### C-001 ‚Äî `$schema` field causes `additionalProperties: false` validation failures

**Affected files:** All YAML governance files that declare `$schema:` at the top (controls, bindings, source entries, the assessment report, waiver template)  
**Root cause:** JSON schemas declare `additionalProperties: false` but do not include `$schema` as an allowed property. The `$schema` field in YAML files is a convenience reference for editor tooling; it is not part of the object definition.  
**Validated:** `ASSESS-PLATFORM-COMPLIANCE-20260708-001.yaml` fails:  
```
Additional properties are not allowed ('$schema' was unexpected)
```

**Fix options (choose one):**  
*Option A (preferred):* Add `"$schema": {"type": "string"}` to the `properties` block of all JSON schemas that govern files which use the `$schema:` field. This is the standard approach ‚Äî JSON Schema itself allows `$schema` as a metadata field.  
*Option B:* Remove `$schema:` lines from all instance YAML files. This removes editor tooling support.

**Decision required before fix.** Recommend Option A.

**Affected file:** `09-assessments/reports/platform-compliance/ASSESS-PLATFORM-COMPLIANCE-20260708-001.yaml`

### C-003 ‚Äî Mapping files use array-wrapper format; `mapping.schema.json` expects single objects

**Affected directory:** `05-mappings/mappings/` (all 10 files)  
**Problem:** Every mapping file uses this structure:
```yaml
source_id: SRC-...
domain: SRC
mappings:
  - id: MAP-...
    source_id: ...
    control_id: ...
```
The `mapping.schema.json` defines a schema for an individual mapping object, not for a grouped file with a `mappings:` array. Running `check-jsonschema` against any mapping file against `mapping.schema.json` will either fail (if `mappings` is an unexpected property) or pass incorrectly (by validating only the top-level wrapper, not the individual mapping objects).

**Fix options:**  
*Option A:* Create a `mapping-collection.schema.json` that wraps an array of mapping objects and validates the grouped files as-is.  
*Option B:* Split mapping files ‚Äî one file per mapping record. File naming: `MAP-{ID}.yaml`. This is consistent with how controls (one file per ID) work.  
*Option C:* Update `mapping.schema.json` to accept the grouped format with a `mappings:` array field.

**Recommendation:** Option A (quickest, no file reorganisation) or Option C (consistent schema change).

**Impact:** The gate criteria file is what the CI workflow actually evaluates. If `CHG-001` is in the gate file but not the profile, the CI will enforce a rule not declared in the profile. This breaks the principle that the gate file is derived from the profile.

**Fix:** Either add `CHG-001` to the profile's `deployment_gate` section with appropriate scope condition, or remove it from the deployment gate file if it was added by mistake.

**Likely intent:** `CHG-001` should be in the merge gate (it already is) but not the deployment gate. Recommend removing from `deployment-gate.yaml`.

## LOW findings

### L-001 ‚Äî `$schema` relative paths in YAML files use inconsistent formats

**Examples:**
- Control files: `$schema: "../../../schemas/control.schema.json"` (relative, 3 levels up)
- Binding files: `$schema: "../../schemas/binding.schema.json"` (relative, 2 levels up)
- Profile file: `$schema: "../schemas/profile.schema.json"` (relative, 1 level up)

**Problem:** Relative `$schema` paths are correct for navigation but are not valid `$id` URIs. JSON Schema validators that follow `$schema` as an absolute URI would fail to resolve these. Most editors (VS Code, etc.) resolve relative paths correctly for editor tooling, so this is primarily a semantic issue.  
**Fix:** Either use absolute URI-like paths (`platform-compliance/schemas/...`) matching the `$id` in the schema files, or keep relative paths and document that they are for editor tooling only.

### L-002 ‚Äî Assessment report `policy_check_id: ""` for manual controls

**Affected:** `ASSESS-PLATFORM-COMPLIANCE-20260708-001.yaml` ‚Äî several entries have `policy_check_id: ""` (empty string)  
**Schema:** `policy_check_id` is an optional field with no minimum length constraint. An empty string is technically valid but semantically confusing ‚Äî it implies a policy exists but is unnamed.  
**Fix:** Either omit the field entirely for manual/not-applicable controls, or add a `minLength: 1` constraint to the schema.
